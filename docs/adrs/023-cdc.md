# ADR 023 - Change Data Capture Implementation

> [!NOTE]
> This ADR is superseded by [ADR 026 - Apache Iggy for CDC](026-apache-iggy-for-cdc.md) due to operational complexities identified with Olake (Temporal and NFS requirements).

## 📝 Context and Problem Statement

To enable real-time analytics and event-driven architecture without impacting production database performance, we need a Change Data Capture (CDC) system that can stream database changes efficiently.

## 💡 Decision

We choose **Olake** as our primary Change Data Capture (CDC) engine.

## 📈 Consequences

- ✅ **Good:** Efficient, log-based data capture with minimal load on source databases.
- ✅ **Good:** Real-time synchronization for analytical workloads and event-driven flows.
- ✅ **Good:** Modern, scalable architecture designed for high-volume data.
- 🚫 **Bad:** New system to manage and monitor in the data pipeline.
- 🚫 **Bad:** Requires specific database configurations (e.g., Logical Replication in PostgreSQL).
